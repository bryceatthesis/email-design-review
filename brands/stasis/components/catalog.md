# Stasis Component Catalog

Index of `brands/stasis/components/` — the Stasis section library, now **60 components** (updated 2026-06-18): **18 approved** + **42 in-review proposals**. The 18 approved are verbatim markup from a Bryce-approved build (`designs/welcome-quiz/cc-from-spec/01..05`) or a coupon-normalized send copy (`designs/welcome-quiz/cc-send/01..05`), with only approved values swapped for `{{SLOT}}` markers. The 42 newly-promoted components are tagged in their own file headers **PROPOSED 2026-06-18 (Bryce design review) — in-review; locks verbatim on first approved-build ship** — design-reviewed and built to `brands/stasis/tokens.json` + Figma mining, but NOT yet approved or shipped, so not yet locked (promotion is not approval); their HTML freezes to verbatim when each first ships in a Bryce-approved build. Each component file carries its own full header (slots with approved values, media classes verbatim, dependencies) — **the file header is the authority; this catalog is the index.**

Slot markers are `{{UPPERCASE_WITH_UNDERSCORES}}` (no inner spaces). Anything in a fragment shaped like `{{ organization.name }}` or `{% coupon_code %}` is Klaviyo Liquid, not a slot — keep it byte-exact.

## Locked vs flexible

Summarized from `usage-profiles.json` (internal CreativeOps working file, outside this public pack) (cross-source usage mining across the Stasis email corpus). Each of the 60 components falls into one of five buckets by what is *invariant* about it; the workhorses are the components whose treatment legitimately flexes per send.

- **Colorway-locked** — the surface color is the constant; content flexes: `promo-strip-yellow` (yellow), `chartreuse-callout-panel` (lime box), `countdown-timer` (dark #0E1A38/#000000 band), `wordmark-row-white` (white-on-dark, BG ∈ #011D59/#0E1A38/#000000), `pull-quote` (yellow/cream testimonial card), `coupon-code-row-yellow` (yellow), `chart-infographic` (beige/chartreuse, function+colorway), `promo-band-black` (black §6.3 band), `stats-panel-dark` (dark/black panel, image varies), `stats-panel-light` (#F7F7F7 panel), `two-col-list` (lime panel), `testimonial-card-centered` (lime/peach/teal closed set).
- **Image-locked** — tied to a specific subject/photo class, exact shot rotates: `unboxing-kit` (the kit), `feedback-reply-block` (rotary-phone "We're All Ears"), `product-duo-image` (the Day+Night pair), `product-shot` (one product), `social-icons-row` (IG/FB/TikTok icon set), `science-spotlight-card` (ingredient/botanical photo), `ingredient-spotlight-card` (botanical/food photo), `testimonial-card-photo` / `testimonial-card-photo-cream` (customer photo, §8.5).

## Disambiguation

Two pairs/sets of components read similarly enough that the keyword mining flagged them as possible duplicates. They are **distinct** and must not be collapsed:

- **`pricing-cards-2col` vs `pricing-tier-cards`** — NOT duplicates. `pricing-cards-2col` is **two equal 50/50 cards** side by side (a head-to-head comparison: SAVE eyebrow, dollar amount, term, SAVE NOW pill per card; pastes as a *panel row* inside a §6.1 lime panel). `pricing-tier-cards` is **three 33/33/33 supply tiers** — an anchored 1/2/3-month ladder with struck was-prices, a % OFF pill, now-price, and a **BEST VALUE tab** on the top tier (pastes as a *row* into the 600px wrap). The usage-profile mining returned near-identical example sets for the two labels; that is an artifact of the keyword matcher seeing "pricing card / strikethrough / BEST VALUE" in both — the *layouts* (2-up comparison vs 3-up ladder) are different modules. Pick by structure: head-to-head = 2col, supply ladder = tier.
- **The hero trio `hero-art-overlay` / `hero-color-band` / `hero-orbit-art`** — three distinct heroes that differ ONLY by background treatment, not by copy/CTA structure (the keyword matcher swept the whole hero region and could not separate them — treat the inflated, overlapping usage counts as a single hero region, and choose by treatment):
  - `hero-art-overlay` = live text/wordmark composited **over a recomposed photographic/product image** (text-free BG art + VML rect fallback).
  - `hero-color-band` = live text reversed onto a **solid color field** (navy/royal/lime/creme), no photo.
  - `hero-orbit-art` = live text above a **floating orbit/constellation product graphic** on a deep navy or black field.

## Usage rules

1. **Paste, fill, don't redraw.** Components are extracted verbatim from Bryce-approved builds. Use one by pasting the fragment at its documented paste level, filling only the `{{SLOT}}` markers, and copying the listed 480px media rules verbatim into the host's `@media only screen and (max-width:480px){ … }` block. The host 600px `.wrap` table must also carry the mobile rule `.wrap{width:100% !important;max-width:100% !important;}` in its `@media (max-width:480px)` block (the universal host-shell prerequisite — components assume it). The host document must carry the light-only pin: both metas (`<meta name="color-scheme" content="light only">` + `<meta name="supported-color-schemes" content="light only">`) and the `:root, body { color-scheme: light only; }` rule.
2. **Non-slot markup is frozen.** Editing anything outside a slot — paddings, colors, weights, VML geometry, conditional-comment structure, pinned copy — is a design change and requires Bryce review before it ships. The pinned values exist because they were measured against approved renders.
3. **No component fits? Propose, don't improvise.** If a layout maps to no catalog entry, propose a new component to Bryce; extract it into this library only after an approved build contains it. Never improvise a one-off section inline.
4. **Colorways are closed sets.** The colorway / value-set tables in component headers (testimonial colorways, wordmark BG+padding pairs, cta-pill geometry variants, pill backgrounds) are the approved palette. Fill every slot from one row; a new color, size, or variant value is a Bryce call, not a fill-in.
5. **Net-new mapping starts here.** Section mapping for net-new emails (email-design Step 2) happens against this catalog: map each section of Bryce's copy to a component row above, and flag anything unmapped as a rule-3 proposal or an imagery-sourcing dependency before directions are drafted.
